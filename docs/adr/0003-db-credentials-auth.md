# ADR 0003 — 경량 자작 쿠키 세션 인증 (비밀번호 없음, OAuth 백로그)

- 상태: 채택(Accepted)
- 날짜: 2026-06-22

## 맥락

지인 그룹용 캐주얼 앱. 진입 마찰을 최소화하고 인증 구현도 단순하게 가져가고 싶음. 로그인은 **이름만 입력 → User 생성·조회 → 로그인**, **비밀번호·이메일 없음**(이름이 곧 신원, "일단은").

> 비밀번호를 빼는 이유: 사용 대상이 지인 그룹뿐이라 진입 마찰을 0에 가깝게. 비번/이메일 입력이 생기면 "귀찮아서 안 쓴다"가 더 큰 리스크. `email`은 선택 필드로 스키마에 남겨둠(추후 이메일 로그인 대비).

## 결정

**Auth.js 대신 경량 자작 쿠키 세션을 쓴다.** Next.js 공식 인증 가이드의 "stateless session" 패턴 그대로:

- `jose`로 세션 페이로드(userId)에 서명 → **HttpOnly 쿠키**에 저장.
- 로그인은 **Server Action**: 이름/이메일 폼 → User upsert → 세션 쿠키 설정.
- 세션 읽기는 서버에서 쿠키 검증(DAL 함수 `getUser()`).

### 왜 Auth.js를 안 쓰나 (원안에서 변경)

- Auth.js의 핵심 가치인 **OAuth/소셜 로그인은 백로그**라 지금 이득이 없음.
- Credentials provider는 본래 **비밀번호 검증용** → 비번 없는 흐름엔 우회가 필요해 오히려 복잡.
- next-auth v5는 beta라 Next 16 호환 리스크.
- Next 공식 문서가 우리 케이스(폼→Server Action→쿠키)를 그대로 안내 → `jose` 한 개면 충분.

## 결과

- 장점: 의존성 최소(`jose`), 비번 관리 흐름 불필요, 진입 마찰 최소, Next 문서가 직접 보장하는 패턴.
- 단점: 비밀번호가 없어 **남의 이름을 치면 그 계정으로 로그인 가능**하고 **동명이인은 같은 계정을 공유**. 지인 신뢰 범위 내에서만 수용("일단은"). 공개 확장 시 이메일/OAuth로 교체.
- **OAuth가 필요해지는 순간**(공개 배포·확장) Auth.js 도입을 재검토. 그때 이 ADR을 대체하는 새 ADR로.
